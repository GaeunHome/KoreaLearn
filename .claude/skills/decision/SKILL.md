# /decision — AI 時代技術決策框架

## 核心前提

AI 已經解決了大部分寫程式碼的難度。**「能不能做到？」** 和 **「難不難？」** 不再是技術選型的主要考量。

**真正的成本在後面**：維護、遷移、安全修補、技術死亡後的重寫。決策應優化**最低長期總成本**。

## 適用場景

| 場景 | 適用？ |
|------|--------|
| 選擇框架、服務、工具或套件 | 是 |
| 架構方式比較（Monolith vs Microservice、SSR vs SPA） | 是 |
| 是否引入新依賴 | 是 |
| Build vs Buy | 是 |
| 升級 vs 維持現狀 | 是 |
| 技術已決定，純實作 | 否 — 直接做 |

## 決策流程

### 步驟 1 — 列出選項

列出所有候選方案。每個一句話。

### 步驟 2 — 假設審查

評估前，明確列出所有隱含假設：

```
假設：
1. [假設] — 證據：[支持依據] | 風險：[如果錯了會怎樣]
2. [假設] — 證據：[支持依據] | 風險：[如果錯了會怎樣]
```

### 步驟 3 — 四維度評分

| 維度 | 選項 A | 選項 B |
|------|--------|--------|
| 長期技術債 | ? | ? |
| 可擴展性 | ? | ? |
| 安全性 | ? | ? |
| 存活風險 | ? | ? |

評分：`低` / `中` / `高` 風險。

### 步驟 4 — Pre-Mortem（預想失敗）

> 假設這個決策在 6 個月後徹底失敗了。什麼出了問題？

```
Pre-Mortem [選項 X]：
1. [失敗模式] — 可能性：[高/中/低] | 爆炸半徑：[小/大/致命]
2. [失敗模式] — 可能性：[高/中/低] | 爆炸半徑：[小/大/致命]
```

### 步驟 5 — 建議

```
建議：選項 X
原因：[一句話]
最大風險：[最大的單一風險]
退場計畫：[如果需要遷移怎麼辦]
遷移成本：[小時 / 天 / 週 / 全面重寫]
```

## 常見陷阱

| 陷阱 | 正確思維 |
|------|---------|
| 「這個比較簡單」 | AI 讓實作難度趨近相等 — 不應成為決定因素 |
| 「先用這個，之後再換」 | 計算「之後再換」的真實成本 — 通常是預估的 3-5 倍 |
| 「大家都在用」 | 流行 ≠ 穩定 — 檢查實際維護狀態 |
| 「功能最多的那個」 | 更多功能 = 更大攻擊面 + 更難升級 |
