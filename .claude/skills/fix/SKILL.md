# /fix — Bug 修復工作流程

## Skills 載入規則

| Skill | 載入時機 |
|-------|---------|
| **debug** | 總是 — 修復前必須系統性調查根因 |
| **testing** | 總是 — 重現並驗證修復的核心 skill |
| **principles** | 根因涉及設計問題（SRP 違反、缺少 DI、耦合過高） |
| **done** | 總是 — 確認測試通過且無回歸 |

## 步驟 1：診斷根因

Bug 描述：$ARGUMENTS

閱讀相關檔案，追蹤執行路徑。

- 使用 `git log --oneline -20` 確認 bug 何時引入
- 使用 `git diff <commit>^ <commit>` 檢查可能導致問題的變更
- 分類 bug 類型：
  - **邏輯錯誤** — 條件錯誤、off-by-one、演算法不正確
  - **Null 問題** — 邊界缺少防護
  - **型別不匹配** — 外部資料的形狀不如預期
  - **競爭條件** — async 操作順序錯誤
  - **設計壞味道** — 深層結構問題的表面症狀
  - **架構違規** — 跨層呼叫導致的問題

用一句話記錄：「Bug 在 [commit/file] 被引入，因為 [根因]。」

## 步驟 2：載入 Skills

套用上方載入規則。總是載入 `testing`。根因涉及結構問題時載入 `principles`。

## 步驟 3：調查

如果 bug 涉及第三方套件或框架：

- 使用 WebSearch 查詢官方文件、changelog 或遷移指南
- 確認建議的修復方式後才開始寫程式碼

如果 bug 純粹在應用程式邏輯中（無外部依賴），跳過此步驟。

## 步驟 4：計畫模式 — 設計重現測試

進入計畫模式。設計最小的測試案例來重現 bug。

- 只寫最小重現案例 — 不要加額外場景
- 明確指定：測試檔案位置、測試名稱、斷言內容、測試資料
- 列出使用者需要確認的問題

**等待使用者確認後才開始寫程式碼。**

## 步驟 5：Red — 寫失敗測試

寫重現測試。執行 `dotnet test` 確認失敗。

- 確認：「測試以 [具體錯誤訊息] 失敗，因為 [根因]。」
- 不要動 production code。
- 如果測試沒有失敗，重現方式有誤 — 回到步驟 1。

## 步驟 6：Green — 最小修復

寫最少的程式碼讓失敗測試通過。

- 不要在此步驟重構
- 不要修不相關的問題
- 確認：「所有測試通過。」

## 步驟 7：Refactor

清理修復和直接受影響的程式碼。

- 改善命名、結構和可讀性
- 確認：「測試仍然通過。」

## 步驟 8：回歸檢查

執行受影響模組的測試套件。

- 如果有更廣的測試套件，一併執行
- 確認：「無回歸問題。」

---
